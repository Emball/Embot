## Remote Debugging & Testing Workflow

NEVER load Embot.py on Windows for testing. The live bot runs on the Linux laptop 24/7. Running a duplicate instance on Windows would register two bots at once and cause issues for the deployed Linux server. Running final tests on the Linux server ensures it utilizes the real world data and it doesn't fail on scenarios it would otherwise succeed on with blank databases and configs. The remote_debug module runs a lightweight HTTP API inside the bot on Linux. When invoked standalone (`python modules/remote_debug.py <command>`), it acts as a client connecting to that API over the local network. Dual-purpose. The remote debug server and client share `config/remote_debug.json`. The token is auto-generated by the server on first startup and stored in the same config file. `"server": true` starts the HTTP server; `"server": false` lets the CLI client connect to a remote server using the same `host`/`port`/`token` values.

REMOTE DEBUG EXEC IS READ-ONLY — NEVER use `exec` to edit code on the server. The server runs the live git repo; editing files directly with sed/echo/etc creates uncommitted changes that block `git pull --ff-only` on the next update, causing sync failures. All code edits must be made in the local repo at `C:\Users\Embis\Documents\Embot`, committed, and pushed — then the server pulls the clean commit. Use `exec` only for reading files, checking logs, or running read-only commands. Exceptions require explicit user approval.

## Versioning & Git

- GitHub token lives in the auth file inside the project (`config/auth.json`).
- Version format: `MAJOR.MINOR.PATCH.MICRO` (e.g. `34.4.0.4`).
- The agent handles all versioning and git operations automatically.
- Version.py increment thresholds (based on total lines changed since last version bump commit):
  - `>= 300-500` lines → **MAJOR** bump (M.m.p.m)
  - `>= 100-150` lines → **minor** bump (m.M.p.m)
  - `>= 20-50` lines → **patch** bump (m.m.P.m)
  - `>= 1-5` lines → **micro** bump (m.m.p.M)
- Increment the version file on every change.
- Commit message = version number only.

- Keep `requirements.txt` synced to actual imports after every edit.

### Cross-Module Dependencies

- `messages.py` is shared state — no bot dependency, imported directly by mod_core and vms_playback.
- `mod_core.py` provides `is_owner()` used by music_archive, community, links, mod_logger (lazy imports inside handlers).
- `mod_suspicion.py` provides `is_flagged()` used by music_archive (lazy import inside handler).
- `mod_core.setup()` is the central hub — creates `ModerationSystem`, imports and wires all other mod modules, registers commands and listeners.
- `vms_core.py` dispatches `vm_transcribed` event; `mod_core` listens and handles automod scanning — VMS has no knowledge of moderation.
- Modules attach themselves to `bot` via attributes (e.g. `bot.ARCHIVE_manager`, `bot._mod_system`, `bot._community_system`, `bot.remote_debug_server`, `bot.vms_manager`).
- `bot.logger` (ConsoleLogger) is available to all modules — set by `Embot.py`.
- `_utils.py` provides `atomic_json_write()`, `migrate_config()`, `script_dir()`, `_now()` — used broadly across modules.

### Embot Startup Flow

1. Parse CLI args
2. Load `config/embot.json` (auto-create defaults if missing)
3. Init `discord.ext.commands.Bot` with `!` and `?` prefixes
4. Create ConsoleLogger (session-scoped log in `logs/`)
5. `on_ready`: load `_version.py`, start console + heartbeat + auto-update loop, call `load_modules()`, sync slash commands
6. Auto-update: `git fetch`, compare version, fast-forward merge, restart on exit code 42 (logs only on update or error)
